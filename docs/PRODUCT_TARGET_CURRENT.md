# 《问道长生》当前产品目标

本文件是当前开发、自动验收与发布决策的共同标准。旧版界面、既有测试通过记录和现有占位素材都不能代替本目标。

## 核心体验

- 玩家进入后应立即理解：自己是谁、身处何种局面、当前章节发生了什么、接下来能够做什么。
- 全流程应像一部长篇修仙小说：主线顺畅、章节有悬念、选择有情绪反馈，不得退化成功能大厅、按钮墙或无目的随机点击。
- 修炼、战斗、秘境、轮回和资源操作只在剧情需要时出现。战斗不得永久开放；秘境必须由具体线索触发；轮回只能在死亡结算后按规则触发。
- 作者编排的主流程不得依赖 AI 随机事件。AI 如保留，只用于摘要、检查或不打断主线的辅助能力。

## 分支与回响

- 宏观锚点和关键相遇可以稳定，但抵达方式、人物立场、关系、资源、承诺、债务、能力代价、战斗条件和结局必须随选择改变。
- 每个关键选择应在随后一至两个章节内通过正文、人物反应或世界状态产生可感知回响。
- 选择按钮只描述角色当下采取的行动，不直接展示“可预见后果”“未知风险”、数值收益或道途攻略。
- 分支可以重新汇流，但汇流后的正文和状态必须记得玩家如何抵达，不能抹平既有关系与代价。
- 失败应生成逻辑完整、可以继续游玩的故事路线，而不仅是数值扣减或重复按钮。

## 参考与原创边界

- 指定研究材料：`D:\work\wanxiang.txt`、`D:\work\dazhuzai.txt`、用户提供的 `https://github.com/ltd0102/ghs/tree/main`，以及《盗妖行》《琼明神女录》《武动乾坤》《斗破苍穹》等同类作品的公开信息。
- 可以研究和借鉴节奏、冲突模型、人物弧线、文风特征、分支汇流、身份落差、期限、契约和情绪设计。
- 发布内容必须重新构思并独立创作；不得逐句复制正文、照搬专有角色，或复刻足以辨认来源的具体桥段。
- 无许可证或权利不明的电子书仓库只能用于研究，不能成为可再分发文本或素材来源。
- 战斗还要参考成熟的长线产品设计：公开可研究的《世界弹射物语》/World Flipper、社区 Starpoint 服务端模拟器（<https://github.com/duosii/starpoint>）及其国服分支（<https://github.com/DontBeAlarmed/startpoint-cn>）只用于观察内容分层、战斗预警、短循环反馈、结算与存档恢复等产品方法；不复制其客户端代码、素材、协议或未授权数据。服务端 GPL 代码与游戏原始资源不自动授予本项目素材再分发权。
- 指定研究 [libnoname/noname](https://github.com/libnoname/noname) 的事件生命周期、技能触发/过滤/代价/效果分层、独立 AI 评价、模式与扩展组织；当前仓库虽附 GPLv3 正文，README 另有非商业请求，且大量三国杀角色、卡牌、语音、美术的再分发权不能仅凭代码仓库推定，因此不引入其代码、数值、文本或资产，只原创转化可组合规则和工程原则。
- 不把两三个知名项目当成研究终点。每次核心玩法重构至少横向核对三类成熟开源产品：战役与叙事状态、短循环与敌人身份、内容包与长期扩展；研究结论必须落成数据结构、交互规则、遥测或测试门禁，而非只写参考名单。
- “去 AI 味”是正文验收项：减少模板化总结、同构三句式和替玩家解释主题的句子；用具体动作、对白、环境和人物反应承载因果，同时保留后台可审计状态。

## 成年关系与黑暗路线

- 玩家和所有亲密关系参与者必须明确为十八岁以上成年人。
- 可以写自愿恋爱、欲望、双修、炉鼎关系和含蓄亲密氛围；明确性行为细节前转场。
- 可以存在强迫、囚禁、控制、创伤、表面顺从、创伤依附、立场腐化以及恢复自主后的黑暗选择。
- 必须分别建模真正自愿、表面顺从、创伤依附和受控状态。强迫情境不得被写成色情奖励或爱情证明；恢复自主后的关系变化必须是一次新的自主选择。

## 音频与美术资源

- 音乐和音效必须实际试听，按章节、地点、人物、战斗和情绪使用。来源、作者、具体许可证、下载地址、文件哈希、修改和署名要求必须可追溯。
- 优先采用明确允许商业使用和项目再分发的高质量开放音乐、音效与美术。`CC0`、`CC BY`、`CC BY-SA` 等仍需逐项核对具体版本和义务；排除 `NC`、来源不明或禁止再分发的资源。
- 开源美术可以主动筛选并接入，但必须同时达到产品审美、角色/场景一致性和许可证要求；质量不足时宁可保持无图事件，不使用重复图、拼贴图或低质占位图。
- 不擅自覆盖仍在制作中的确认资产，不把本地 Godot 导入元数据或临时产物误当成可发布美术。

## 自动验收与发布

- 必须通过 Godot 导入与脚本解析、单元/回归、存档迁移、分支可达性、文本重复率、十世长跑、10,000 局试玩模拟、多分辨率真实渲染和 Windows 构建。
- 自动试玩不仅检查崩溃，还应发现死路、逻辑矛盾、重复、节奏失衡、入口常亮、字体难读、选择无反馈和音频不合格。
- 战斗验收必须覆盖多种玩家策略和构筑，记录动作熵、最长重复、无效选项比例、敌人签名机制触发、第二阶段到达、战技兑现和胜败/撤离分布；单一自动策略的零崩溃不能证明玩法有趣。
- 发布前清理缓存、构建产物、临时下载和无用资源，核对 `.gitignore`、仓库体积、许可证清单与最终差异。
- 非美术核心体验验收通过后，用 Godot 新版完整替换 GitHub `main`；明确记录尚待接入的美术缺口，但不得用美术缺口掩盖玩法、文本、界面或音频问题。

# 普通战斗产品参考与实现边界

更新日期：2026-07-22。

## 可借鉴的产品原则

1. 每场战斗先回答“为什么打”。敌人身份、追来的原因和失败赌注必须在首屏可读，不能只埋在滚动日志里。
2. 每一回合先回答“眼前要处理什么”。敌方意图、伤害范围和附带状态必须同时出现；各行动只说明自身收益和代价，不直接替玩家标出标准答案。
3. 普通行动要汇成短循环。连续读懂敌意需要形成可见进度，并在数回合内兑现一次明确爆发；正确接法之外应保留带代价的稳妥选择，避免把战斗退化成唯一答案的按钮问答。
4. 胜利、撤离和失败都是剧情分支。结算必须保存来源选择与余波；撤离回到主线，战败进入既有轮回流程，任何结果都不能留下常驻战斗入口或阻断存档。
5. 自动战斗、存档恢复和长局模拟必须调用同一规则，不为演示另造一套结果。

## 本项目的原创实现：三拍破势

- 每种公开敌意都有一项能推进破势的对应行动：迅击用斩击抢先，蓄势重击用守势承接，结印护身用术法穿透，撕裂经脉用守势减伤，蚀心咒用术法反压；部分意图还允许不推进节拍但能保住局面的代价行动。界面以“进势 / 保拍 / 断势”描述真实后果，不显示“推荐 / 备选”。
- 连续三次用对应行动接住敌意，形成一次“破势”。下一次斩击或术法获得 `30%` 威力增幅，点击前的伤害预测已包含增幅；使用后清空并可从这一手重新积势。
- 服丹和脱战不伪装成应对行动，也不会凭空增加进度。完全不合适的行动会打断当前节拍，但带代价的备选可以保住当前进度；本场最佳连势、备选使用和破势消费都进入战史遥测。
- 普通敌人在气血过半时会显出原创的第二相，改变后续意图节奏；战斗标题下直接显示“此战所争”，中央战术区同时显示三拍进度、当前危险、三步意图和行动收益。
- 剧情选择只登记迎战意图；夺回证据、击退追兵等成功事实在真正胜利后才写入。撤离、战败和敌踪过期分别结算自己的持久余波，且同一战斗结果不能重复结算。
- 这套机制只使用本项目既有数据和 Godot 代码，没有引用 StarPoint、官方客户端或官方资产。

## 下一层产品门槛

- 装备、道途和剧情承诺最终应改变可用招式或破势兑现方式，而非只增加攻防数值。
- 撤离应逐步转为玩家可制造的脱身窗口；奖励不得随拖延回合数增长。
- 长程验收除稳定性外，还要覆盖多种代理策略、动作熵、重复动作、二相到达率、破势兑现与构筑胜率，不能用单一自动策略证明“有趣”。

## 发布门禁

- 战斗入口只能由有期限且带来源的敌踪开启，进入后由战斗系统原子消费敌踪。
- 意图预测、签名状态、破势进度、存档恢复和实际伤害必须使用同一公式。
- 撤离后角色仍可继续，余波进入战史；战败必须正常进入轮回，不出现死屏。
- 对方仓库的客户端补丁、协议实现、数值表、CDN、文本和美术不得进入构建产物。

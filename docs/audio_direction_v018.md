# v0.18 产品级音频方向

## 目标

《问道长生》的声音不是持续铺满的背景装饰，而是世界状态、人物行动和玩家选择的第二条信息通道。成品需要在长时间游玩中保持清晰、克制和可辨认：重要反馈能在混合中立即被听见，不重要的层主动让位，静默也被当作叙事手段。

本规范先以古典修仙纪完成可发布的纵向切片，再把同一事件契约扩展到其余五个时代。时代差异必须来自乐器、节奏、空间、材质和交互反馈的完整组合，不能只给同一首曲子更换均衡器或增加环境噪声。

## 当前基线

- 当前仓库已经建立 `default_bus_layout.tres`、常驻 `AudioDirector`、独立设置页、版本化授权清单和 111 个可复现的六纪元原创运行时素材；每个纪元都有独立的探索、压力、决战三态配乐与环境底床，出牌、命中与护体各有 4 个独立时代变体，普通战斗、秘境、首领破相、结算与轮回已消费结构化声音事件。
- 18 条流式 Ogg 配乐采用统一的 64 秒、120 BPM、32 小节和采样边界；`menu/world/event/reincarnation` 映射探索态，`combat/dungeon` 映射压力态，`boss` 映射决战态。双音乐声部从当前播放相位进入并进行 1.75 秒交叉淡化，纪元切换沿用同一同步契约。
- 当前素材已通过 48 kHz 运行时格式、SHA-256、峰值、RMS、DC 偏移、跨纪元/跨状态内容差异、Ogg 页与最终 granule、循环接缝和 Godot 实际解码门禁，并按项目发布政策标记为 `final`；`build_godot.ps1 -ProductRelease` 会主动拒绝任何重新混入的 `prototype_only` 或 `production_candidate`。
- 地点级声景、天气与稀疏点声，以及其余五时代的首领/轮回等低频专属终稿仍未完成；现阶段共享低频语义会明确回退古典终稿，不把单条环境循环冒充完整地点声景。
- 当前玩法已经能返回普通战斗、秘境出牌、护体、受击、压力、心魔、首领破相与结算等结构化结果；声音应消费这些结果，不从按钮文字、提示文案或画面颜色反推玩法状态。
- 第一阶段不得把临时素材当成最终素材提交。占位音频若确有开发需要，必须明确标为 `prototype_only`，且发布验证应拒绝打包。
- 游戏必须在完全静音时仍然可理解、可操作和可通关；音频只增强反馈，不独占规则信息。

## 总线架构

Godot 项目使用受版本控制的 `res://default_bus_layout.tres`。页面或玩法脚本不得自行创建总线、修改总线增益或直接连接到 `Master`。

```text
Master
├─ Music
├─ Ambience
├─ SFX
├─ UI
└─ VO
```

| 总线 | 职责 | 默认相对增益 | 约束 |
| --- | --- | ---: | --- |
| `Master` | 最终输出、全局限幅和动态范围模式 | `0 dB` | 只放最终保护处理，不用它修正单个素材 |
| `Music` | 乐曲、分层乐段和叙事尾奏 | `-3 dB` | 必须流式播放；时代切换按乐句或小节过渡 |
| `Ambience` | 常驻声景、天气、地点和稀疏环境点声 | `-5 dB` | 至少包含宏观底床与低频率点声两层 |
| `SFX` | 世界、战斗、能力、心魔和首领反馈 | `-1 dB` | 关键玩法反馈优先级高于环境点声；未来内容量需要时再增加子总线 |
| `UI` | 确认、返回、禁用、页签和通知 | `-4 dB` | 短、低疲劳；自动聚焦和界面重建不得发声 |
| `VO` | 未来的角色语音、呼吸与明确的人声叙事 | `0 dB` | 当前无配音也保留；不得把非人声塞入此总线 |

- `Music` 和 `Ambience` 在首领显形、纪元揭示、心魔显化及未来语音期间接受侧链让位；常规 UI 点击不得触发全局 ducking。
- ducking 的衰减通常为 `2 至 4 dB`，攻击 `20 至 80 ms`，释放 `250 至 700 ms`。首领或纪元转场可短时达到 `6 dB`，但不得产生明显抽吸。
- `Master` 默认只允许透明保护性限幅，输出上限为 `-1 dBTP`。素材、总线和事件层级不合格时必须在源头修正，不靠最终限幅器持续削峰。
- 用户设置至少提供主音量、音乐、环境、音效、界面和语音六个独立滑杆；每类可调到 `0`，另提供全局静音，取值持久化到独立设置文件，不写入角色存档。

## AudioDirector

`AudioDirector` 是主场景下跨页面常驻的唯一音频入口；未来拆分多场景时可无缝迁移为 Autoload。除它管理的播放器外，不允许页面、按钮、玩法系统或临时反馈节点直接调用 `AudioStreamPlayer.play()`。

核心职责：

- 加载并校验音频 manifest，将稳定事件 ID 映射到资产、变体、总线、优先级、冷却和并发策略。
- 持有音乐、声景、UI、世界音效与战斗音效播放器池，避免频繁创建节点。
- 根据 `era_id`、屏幕、地点、战斗位阶、压力、气血、结果和无障碍设置选择声音，不改变任何玩法结果。
- 管理音乐分层、无缝循环、按小节切换、场景淡入淡出、转场尾音和应用失焦策略。
- 对相同事件做防抖、并发限制、变体轮换和优先级抢占，避免快速操作形成刺耳叠加。
- 提供测试模式：固定变体种子、禁用随机时间点声、记录已解析事件，不依赖系统时钟得到可重复结果。
- 在资源缺失或音频设备不可用时静默失败并记录一次可定位错误，绝不阻塞输入、存档或玩法推进。

建议的稳定接口：

```gdscript
AudioDirector.play_event(event_id: String, context: Dictionary = {}) -> bool
AudioDirector.set_context(context_id: String) -> void
AudioDirector.set_bus_percent(bus: String, value: float) -> void
AudioDirector.set_setting(key: String, value: Variant) -> void
AudioDirector.get_settings() -> Dictionary
AudioDirector.save_settings() -> void
```

- `play_event` 只接受 manifest 已登记的事件；未知 ID 在开发构建中报错，在发布构建中限频记录并返回 `false`。
- 页面销毁、退出秘境或读档时必须停止对应 `scope_id`，禁止上一页面的循环、尾音或延迟事件泄漏到新状态。
- 全局同时发声的普通 voice 建议硬上限为 `32`：UI `4`、战斗 `12`、世界 `6`、环境点声 `6`，音乐/环境底床保留 `4`。达到上限时先淘汰低优先级、最旧且感知最弱的声音，不截断纪元、轮回、首领阶段或关键受击提示。
- 同一短音效默认至少准备 `3` 个合格变体；高频事件使用 shuffle bag 避免连续重复。音高随机仅用于非音高性材质，通常不超过 `±2%`；音乐、提示音程和人声禁止随机移调。

## 事件契约

玩法系统只发布“发生了什么”，不指定具体文件、音量或音色。事件 ID 使用小写点分层级，并在发布后保持向后兼容；资产可以替换，事件含义不能悄悄改变。

每次请求的标准上下文如下：

| 字段 | 类型 | 必填 | 含义 |
| --- | --- | --- | --- |
| `event_id` | `StringName` | 是 | 例如 `dungeon.card.cast` |
| `era_id` | `StringName` | 是 | 六个稳定时代 ID 之一 |
| `scope_id` | `StringName` | 是 | 页面、战斗或转场实例，用于生命周期清理 |
| `source_id` | `StringName` | 否 | 角色、能力、敌人、物品或按钮的稳定 ID |
| `result` | `StringName` | 否 | `success`、`failure`、`blocked`、`critical` 等语义结果 |
| `intensity` | `float` | 否 | 归一化到 `0.0 至 1.0` 的表现强度，不传原始伤害当音量 |
| `position` | `Vector2` | 否 | 画面内声像依据；关键 UI 和叙事反馈保持居中 |
| `tags` | `PackedStringArray` | 否 | 如 `boss`、`elite`、`heart_demon`、`second_phase` |
| `seed` | `int` | 否 | 回放和测试所需的确定性变体种子 |
| `dedupe_key` | `StringName` | 否 | 同一帧或短窗口内合并重复反馈 |

- `context` 只能影响表现选择，不能写回玩法状态或成为奖励、命中、死亡和存档判定依据。
- 强度先经过每个事件在 manifest 中声明的曲线和上下限，再影响层数、变体或极小范围增益；不得直接把 `intensity` 转换为线性音量。
- 一次动作产生多条结构化结果时，由拥有该动作语义的适配层发布有限事件。例如攻击被护体完全吸收时发布 `combat.guard.absorb`，不能同时无条件叠加完整 `combat.hit.flesh`。
- UI 事件依据控件语义发布，不从鼠标按下统一播放：确认成功用 `ui.confirm`，校验拒绝用 `ui.invalid`，返回用 `ui.back`，禁用控件不伪装成成功。

首批稳定事件至少包括：

| 领域 | 事件 ID |
| --- | --- |
| UI | `ui.focus`、`ui.confirm`、`ui.back`、`ui.invalid`、`ui.tab.change`、`ui.notice` |
| 修炼与世界 | `cultivation.begin`、`cultivation.pulse`、`cultivation.breakthrough.success`、`cultivation.breakthrough.failure`、`world.event.reveal`、`world.era.transition` |
| 普通战斗 | `combat.attack.cast`、`combat.hit.flesh`、`combat.guard.gain`、`combat.guard.absorb`、`combat.player.hurt`、`combat.enemy.defeated`、`combat.victory`、`combat.defeat` |
| 秘境 | `dungeon.enter`、`dungeon.route.select`、`dungeon.card.cast`、`dungeon.pressure.rise`、`dungeon.heart_demon.manifest`、`dungeon.elite.reveal`、`dungeon.boss.reveal`、`dungeon.boss.phase_break`、`dungeon.run.complete`、`dungeon.run.withdraw` |
| 轮回与存档 | `reincarnation.begin`、`reincarnation.legacy.retain`、`reincarnation.complete`、`save.success`、`save.failure`、`load.success`、`load.failure` |

## 六时代声景

所有时代共享可识别的《问道长生》核心动机：一个不超过五音的“旧玉回声”。它可以改变音色、速度、和声和空间，却不能在每个时代都以相同旋律、相同乐器直接复制。跨世记忆出现时，旧时代层可以短暂渗入当前声景，随后回到当前时代的混音规则。

| 时代 | 音乐与节奏 | 环境与材质 | 交互/战斗辨识 | 禁止方向 |
| --- | --- | --- | --- | --- |
| 古典修仙纪 `classical` | 古琴、箫、埙、低动态弦鸣；呼吸式自由速度进入清晰但稀疏的脉冲 | 镜湖水、远钟、灯河木舟、布幡与克制市声 | 玉、木、纸、剑气和灵息；攻击留出起势与收锋 | 满铺“古风”乐器、旅游景区环境声、每次点击都敲木鱼 |
| 灵机蒸汽纪 `steam` | 锻锤与机械循环形成节拍，管钟、拉弦和蒸汽呼吸承担旋律 | 炉膛、铜管、轨道、阀门与远处厂房层级 | 金属应力、齿轮咬合、压力释放；不能牺牲命中清晰度 | 通用柴油朋克轰鸣、持续低频遮蔽、机械声无节奏随机堆叠 |
| 星穹道网纪 `star_network` | 玻璃质拨奏、颗粒延音、脉冲与人性化呼吸层并存 | 轨道低鸣、神识节点、遥远数据风与稀疏定位脉冲 | 身份分叉使用空间复制与错位回声，关键提示仍保留实体瞬态 | 廉价科幻哔声、过亮高频、把所有声音加长混响当未来感 |
| 废土返道纪 `wasteland` | 低密度残缺动机、旧器物敲击和不稳定持续音，更多使用静默 | 干风、黑雨、车体与远处火种设施；声景必须有远近变化 | 布、骨、锈蚀金属和有限灵力；强动作短而干，强调资源匮乏 | 连续大风白噪声、电影预告片低音轰击、把稀疏误做成无反馈 |
| 末法裂变纪 `final_age` | 被截断的乐句、失谐泛音、脉搏与逐渐熄灭的旧玉动机 | 灵脉断续、空站结构、余烬和异常寂静窗口 | 施法呈现点火失败、断流和结构裂变；失败声不能惩罚听觉 | 高频耳鸣、突发满幅噪声、用失真掩盖层级、长时间焦虑低频 |
| 仙朝鼎盛纪 `immortal_dynasty` | 编钟、笙、鼓与宽阔弦层形成礼制秩序；越接近失控越暴露机械般精确的节拍 | 浮空城、风廊、远朝仪与庞大建筑反射 | 敕令、命籍、印玺和阵列使用明确的层级落点 | 影视宫廷套曲复刻、持续宏大合唱、把文化气质简化成锣鼓齐鸣 |

- 真实民族乐器录音必须尊重演奏法和音域；没有合格演奏或授权素材时，宁可使用抽象材质设计，也不使用明显失真的廉价音源冒充。
- 每个时代至少拥有探索、压力和决战三种音乐状态，以及地点底床、天气层和稀疏点声三类环境内容；这些层应共享同步规则并能平滑切换。
- 音乐不能对每次普通点击或小数值变化作曲式响应。玩家在主界面长时间阅读时，音乐应允许自然留白，循环周期和变奏足以避免一分钟内被听穿。

## 古典修仙纪纵向切片

古典纵切是其余五时代开工前的质量闸门，不是低保真样板。它需要使用可商用终稿素材覆盖一条完整体验链：标题界面 -> 开始/续档 -> 主界面修炼与世界事件 -> 普通战斗 -> 镜湖秘境普通敌人/精英/首领第二相 -> 结算或撤离 -> 轮回。

最低内容集：

- 一套可分层的古典主题，包含探索底层、压力层和决战层；各层同速、同长、同循环点，支持按小节进入和退出。
- 至少两套地点底床：镜湖/灯河灵市与空阙秘境；每套至少有一个无缝宏观循环和一组低频率、可定位的随机点声。
- UI 核心语义六类齐备，每类至少 `3` 个变体；聚焦声必须足够轻，连续键盘导航不会疲劳。
- 修炼、突破成功/失败、事件揭示、纪元揭示、存取档成功/失败、轮回开始/遗产保留/完成均有独立语义反馈。
- 普通战斗完整覆盖起手、命中、护体获得/吸收、玩家受伤、敌方击破、胜利和失败，不能以一个通用撞击声包办。
- 秘境完整覆盖能力来源差异、出牌、压力上升、心魔显化、精英显形、首领显形、破相、击破、完成和撤离；首领破相必须由前兆、断点和新相落位组成，而不是单个更响的爆炸声。
- 同一高频战斗事件至少 `4` 个可轮换变体；同屏多目标或同帧多段反馈需有合并策略，连续十次操作不出现明显机关枪效应。
- 所有循环提供精确采样级循环点；音乐和环境声连续播放十分钟不得出现空洞、爆音、相位跳变或可识别的尾音截断。

只有古典纵切通过本规范的自动、运行时和导出验收后，才可批量制作其余时代。扩展时优先复用事件契约、播放逻辑和混音目标，不复用会削弱时代身份的终稿素材。

## 素材与运行时格式

- 制作母版统一为 `48 kHz`、`24-bit WAV`；工程保留可回溯的无损母版与编辑会话，但大型会话和未裁剪原录音不直接进入游戏导出目录。
- 音乐和长环境循环在 Godot 中使用流式 `Ogg Vorbis`，导入后复核循环点和实际编码接缝；短 UI 与关键瞬态优先使用无损 WAV，较长非关键 SFX 可使用经解码与响度门禁确认的 Ogg。
- 禁止 MP3 作为循环或交付源，禁止运行时联网下载素材，禁止依赖开发机系统音色库。
- 需要定位的单点音效优先单声道；音乐、宽环境床和确有空间设计的转场使用立体声。所有素材必须检查单声道折叠，不得因相位抵消丢失关键反馈。
- 文件名使用稳定小写 ASCII，例如 `classical_dungeon_heart_manifest_03.wav`；随机导出名、`final_final2`、供应商原文件名和空格不得进入运行时目录。
- 音乐和长循环按需流式加载；短高频音效由 `AudioDirector` 池化。不得为了避免加载管理而在启动时把全部六时代音频解码进内存。

## 混音与响度

响度以最终游戏内捕获为准，而不是只看素材文件的离线归一化。以下数值是验收目标，不是要求所有瞬间持续贴线：

- 默认模式下，十分钟代表性游玩捕获目标为 `-18 至 -16 LUFS-I`，最终输出不得超过 `-1 dBTP`，不得出现数字削波或限幅器持续工作。
- 音乐母版交付参考为 `-20 LUFS-I ±1 LU`、不高于 `-1 dBTP`；环境底床参考为 `-30 至 -24 LUFS-I`。进入游戏后仍需按场景和总线重新平衡。
- 常规 UI 瞬态参考为 `-28 至 -22 LUFS-S`；普通战斗反馈为 `-24 至 -18 LUFS-S`；极少数首领破相或纪元转场可短时达到 `-16 LUFS-S`，但不能连续触发。
- 关键玩法瞬态在音乐与环境同时播放时应保持可辨认，一般领先局部背景 `3 至 8 dB`，优先通过编配、频段和 ducking 留位，不靠无限加响。
- 低频能量必须有目的。非首领 UI、普通脚步、小型命中不得占用大量 `80 Hz` 以下能量；长时间环境低频需在普通耳机和小扬声器上都不过载。
- 左右声像只增强方位，不承载唯一信息。界面、存档、轮回、首领阶段和危险警告等关键反馈保持稳固中心像。
- 默认动态范围适合普通桌面扬声器和耳机；“夜间模式”压缩大动态并收窄强瞬态，“宽动态模式”保留更大音乐和环境起伏。模式切换不得改变事件是否可听见。

## 无障碍与舒适度

- 每类音量可独立调整到 `0`，静音状态清楚可见；重新启动后保持设置。
- 提供“减少突发强音”选项，降低首领破相、纪元转场、雷鸣和失败冲击等峰值，但保持其节奏和语义轮廓。
- 提供“减少高频刺激”选项，替换或柔化耳鸣、金属尖啸、玻璃裂鸣等内容。产品默认不得包含模拟耳损的长耳鸣。
- 提供单声道输出选项；单声道下所有关键事件仍可分辨，响度不异常抬升或塌陷。
- 未来加入语音后，必须提供完整字幕、独立语音音量和可关闭的非语言人声；字幕不由音频文件名临时生成。
- 危险、失败、可操作状态、敌方意图、压力临界和首领转相必须同时有视觉或文本反馈，不能只靠声音提示。
- 快速连续输入时，UI 聚焦与确认声需要防抖；长时间菜单操作、刷牌和重复修炼不得导致听觉疲劳。
- 应用失焦时默认淡出音乐和声景并暂停非必要一次性音效；是否继续后台播放由用户设置决定，恢复焦点时不得补播积压事件。

## 授权 manifest

所有进入 `res://audio/` 的资产都必须先登记到版本化的 `res://audio/audio_manifest_v1.json`。manifest 是构建输入和发布审计依据，不是事后补写的素材表。

每条记录至少包含：

```json
{
  "asset_id": "classical_dungeon_heart_manifest_03",
  "runtime_path": "res://audio/sfx/classical/dungeon_heart_manifest_03.wav",
  "source_master": "audio_source/classical/dungeon_heart_manifest_03_master.wav",
  "role": "sfx",
  "era_ids": ["classical"],
  "event_ids": ["dungeon.heart_demon.manifest"],
  "creator_or_vendor": "",
  "source_url_or_receipt_ref": "",
  "license_name": "",
  "license_url": "",
  "commercial_use": true,
  "redistribution_in_game": true,
  "attribution_text": "",
  "modifications": "edited, layered, mixed",
  "generator_and_version": "",
  "created_or_acquired_at": "YYYY-MM-DD",
  "sha256": "",
  "sample_rate_hz": 48000,
  "channels": 1,
  "loop_start_sample": null,
  "loop_end_sample": null,
  "measured_lufs_i": null,
  "measured_true_peak_dbtp": null,
  "release_state": "final"
}
```

- 自录、委托、素材库、合成器预设和生成式工具都要记录准确来源与权利依据。生成素材需要记录工具、模型/版本、生成日期和当时适用的商用条款；不能仅写“AI 生成”。
- 禁止使用来源不明的网络下载、无法证明商用权利的剪辑、从影视或游戏中提取的声音，以及要求以独立素材形式再分发的许可证。
- attribution 要求必须能从 manifest 生成发布包中的鸣谢和许可证文件。发票、合同原件和个人信息不进入公开仓库，只记录可由项目所有者追溯的内部凭据编号。
- 修改素材不等于获得新版权。每个分层素材都要可追溯，合成事件不能用一条“自制”记录掩盖未经授权的源层。
- `prototype_only`、缺少哈希、权利字段为空、文件哈希不符或许可证不允许商业发行的资产，发布构建必须失败。

## 自动验收

CI 和本地完整构建至少执行以下检查：

1. 校验 `default_bus_layout.tres` 中总线名称、父子关系、默认增益和必要效果；禁止运行时脚本创建未登记总线。
2. 校验 `audio_manifest_v1.json` 的 schema、唯一 `asset_id`、路径存在性、SHA-256、事件引用、时代 ID、授权字段和 `release_state`。
3. 反向扫描 `res://audio/`，任何未登记文件、孤立 manifest 记录、MP3、错误采样率、意外多声道或原始供应商文件都使检查失败。
4. 扫描 GDScript，除 `AudioDirector` 及其测试外不得直接实例化或播放 `AudioStreamPlayer`；所有调用的事件 ID 必须在事件目录登记。
5. 以 headless Godot 导入并加载所有音频资源、总线和 `AudioDirector`，验证缺失资源不会阻断主场景、存档、读档或测试退出。
6. 固定种子回放代表性事件序列，断言事件解析、时代变体、scope 清理、防抖、并发上限、优先级抢占和用户静音设置一致。
7. 对 WAV/Ogg 解码结果检查峰值、DC 偏移、静音头尾、循环点范围和循环接缝；音乐/环境循环进行至少十轮拼接分析，不得出现超阈值瞬态或长度漂移。
8. 离线渲染古典纵切的菜单、普通战斗、秘境首领和轮回四段混音，生成 LUFS-I、LUFS-S、LRA、true peak 和削波报告；超出本规范目标需显式失败，不能只输出警告。
9. Windows 导出包在干净环境后台启动，验证音频设备存在与不存在两种情况；进程均能进入主场景、正常退出且不访问网络。

自动报告必须保存工具版本、输入 manifest 哈希和测量结果，避免不同机器的“通过”不可比较。发布后的实际玩家反馈继续进入后续调音，但不构成发布前签字门槛。

## 发布后体验回收

项目所有者可在发布后组织体验者按以下路径反馈；这些反馈用于持续优化，不阻塞已经通过自动、运行时和导出验收的版本：

- 使用参考级耳机、普通封闭耳机、常见桌面音箱和笔记本扬声器各完成一次关键路径；不得只在制作人的监听环境验收。
- 在默认、夜间、宽动态、单声道、音乐静音、音效静音和全部静音下走完古典纵切；任何模式都不能丢失玩法理解或阻塞操作。
- 连续游玩至少 `30` 分钟，重点记录 UI 高频声、循环暴露、低频疲劳、尖锐瞬态和同一战斗反馈的重复感。
- 盲听普通命中/护体吸收、成功/失败、精英/首领、首领第一/第二相、压力上升/心魔显化；不看画面时应能区分语义类别，但不要求仅凭声音还原数值。
- 逐一检查画面与声音同步：UI 响应应贴合操作，命中声贴合接触，首领破相的前兆/断点/落位与程序化演出一致，结算尾音不得抢先宣告结果。
- 六时代扩展阶段进行响度匹配后的盲听辨识。测试者应能从音乐、环境和交互材质的组合识别时代，而非依赖独白或单一标志音。
- 对所有循环执行十分钟真实播放，监听接缝、编码毛刺、音量呼吸和过早重复；对快速 UI 与多段战斗执行压力测试。
- 核对游戏内鸣谢、许可证文本与 manifest 自动生成结果，随机抽查源文件、合同/订单凭据和哈希。

若反馈发现削波、关键反馈不可辨、静音模式不可玩、明显循环接缝、突发强音不受控或跨页面残留，应建立可复现用例并进入下一补丁；授权缺口仍属于发布前自动硬失败。

## 实施顺序与完成定义

1. 建立总线、`AudioDirector`、用户设置、事件目录、manifest schema 与自动静态校验。
2. 用开发期占位素材接通古典纵切事件，但占位资产保持 `prototype_only`，先验证生命周期、并发、确定性与无声降级。
3. 制作或合法采购古典时代终稿音乐、声景和音效，完成混音、无障碍与自动运行时验收，并删除所有发布路径中的占位素材。
4. 冻结稳定事件契约，以古典切片的测量和听感为基准依次制作其余五时代；每个时代单独通过相同门槛。
5. 六时代完成后进行跨时代连续游玩、轮回和读档恢复验收，确认旧时代尾音、循环和状态不会泄漏，核心动机能形成跨世连续性。

音频系统只有在资产权利完整、所有稳定事件可解析、六时代声景均达到终稿质量、自动报告通过、Windows 干净导出可离线运行时，才算达到产品交付标准；发布后体验反馈继续用于调音迭代。
